# Benchmarks

`@opentf/obj-diff` is built for two things other diffing libraries trade away: **compact diffs** and **correct handling of native JavaScript values**. The numbers below focus on both.

> The benchmark scripts live in [`/benchmarks`](https://github.com/Open-Tech-Foundation/obj-diff/tree/main/benchmarks) in the repository. Run them yourself with `bun benchmarks/category-a-diff.js` (and the other `category-*.js` files). Timings were captured on an Intel Core i7-8700K under Bun 1.4.0 — absolute numbers vary by machine, but the relative gaps are stable.

## Diff size (fewer operations)

Op counts are **deterministic** — they don't depend on the machine. This is where compact array diffs pay off. Positional differs emit one operation per shifted element; `obj-diff` emits one operation per actual edit.

| Edit on a 10,000-element array | `@opentf/obj-diff` | microdiff |
| :--- | ---: | ---: |
| Insert 1 element at the front | **1** | 10,001 |
| Insert 1 element in the middle | **1** | 5,001 |
| Remove 1 element from the middle | **1** | 5,000 |
| Remove a run of 10 elements | **10** | 5,000 |

Smaller diffs mean smaller audit logs, smaller sync payloads, and smaller undo stacks — see [Use Cases](/docs/use-cases).

## Speed

`obj-diff` is competitive on ordinary objects and **dramatically faster on native collections** like `TypedArray`, which most libraries walk element-by-element (or convert to plain objects first).

| Scenario | `@opentf/obj-diff` | microdiff | deep-diff | just-diff |
| :--- | ---: | ---: | ---: | ---: |
| One property changed (small object) | 2.00 µs | 1.39 µs | 1.18 µs | 2.58 µs |
| Mixed object changes | **1.88 µs** | 2.17 µs | 2.07 µs | 3.04 µs |
| Large `TypedArray` (10k, 1 change) | **21 µs** | 4.45 ms | 85.7 ms | 5.87 ms |

On the `TypedArray` case `obj-diff` is roughly **200× faster than microdiff** and thousands of times faster than `deep-diff`, because it compares typed arrays natively instead of treating them as generic objects.

## Honest trade-offs

Benchmarks are only useful when they include the cases you *don't* win:

- **Tiny shallow objects**: for a single primitive change on a small plain object, positional differs like `microdiff` and `deep-diff` are a few hundred nanoseconds faster. The difference is negligible at real-world scale.
- **Plain large numeric arrays**: computing a *minimal* edit script (Myers LCS) costs more than a raw index-by-index scan, so a heavily-changed 10k plain array can be slower to diff than a positional pass — the payoff is the far smaller diff it produces. When the edit distance exceeds an internal cap, `obj-diff` falls back to index-by-index comparison to keep worst-case time and memory bounded. See [Caveats](/docs/caveats).

The takeaway: reach for `obj-diff` when you care about **diff quality and correctness across real JS values**, not just raw nanoseconds on flat JSON.
