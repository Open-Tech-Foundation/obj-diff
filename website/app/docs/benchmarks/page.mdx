# Benchmarks

`@opentf/obj-diff` is built for two things other diffing libraries trade away: **compact diffs** and **correct handling of native JavaScript values**. Every number on this page is generated by [`benchmarks/collect.mjs`](https://github.com/Open-Tech-Foundation/obj-diff/tree/main/benchmarks/collect.mjs) and rendered straight from its JSON output — nothing here is typed by hand.

> Generated **{meta.generatedAt}** on {meta.cpu} under {meta.runtime}. {meta.note}

## Diff size (fewer operations)

Op counts are **deterministic** — they don't depend on the machine. This is where compact array diffs pay off: positional differs emit one operation per shifted element, while `obj-diff` emits one operation per actual edit.

<OpCountTable />

Smaller diffs mean smaller audit logs, smaller sync payloads, and smaller undo stacks — see [Use Cases](/docs/use-cases).

## Speed

Average latency per call, lowest is highlighted. `obj-diff` is competitive on ordinary objects and **far faster on native collections** like `TypedArray`, which most libraries walk element-by-element.

<SpeedTable />

On the `TypedArray` case `obj-diff` is roughly **{taFactor}× faster than microdiff** — and orders of magnitude faster than `deep-diff` — because it compares typed arrays natively instead of treating them as generic objects.

## Honest trade-offs

Benchmarks are only useful when they include the cases you *don't* win. Read the speed table above both ways:

- **Small and deeply-nested objects**: `obj-diff` sits within a whisker of the fastest positional differ. The differences are nanoseconds and irrelevant at real-world scale.
- **Large plain numeric arrays**: computing a *minimal* edit script (Myers LCS) costs more than a raw index-by-index scan, so a positional differ can diff a big plain array faster — the payoff is the far smaller diff `obj-diff` produces. When the edit distance exceeds an internal cap, `obj-diff` falls back to index-by-index comparison to keep worst-case time and memory bounded. See [Caveats](/docs/caveats).

The takeaway: reach for `obj-diff` when you care about **diff quality and correctness across real JS values**, not just raw nanoseconds on flat JSON.
